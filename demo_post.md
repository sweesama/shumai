## 一、Demo 简介

我做了个网站叫「书脉 AI」，简单来说就是：**把中国古籍搬上网，再用 AI 帮你读懂它**。

你可以把它理解成"古籍版的知识管理工具"——不是单纯把书扫描上网（中华典籍网已经做了），而是在原文基础上，用 AI 自动提取每本书里的人物、地点、概念，画成一张可以点着玩的知识关系网。

**面向谁**：
- 学文言文的学生——看原文费劲，需要翻译和注解
- 传统文化爱好者——想知道一本书里几十个人物到底是什么关系
- 任何对古籍好奇但被门槛劝退的人

**三个核心功能**：

**功能一：藏书阁——83 部古籍在线阅读**
收录了从《论语》《庄子》到《山海经》《史记》等 83 部经典古籍，支持全文阅读、章节切换。首页精选《山海经》，配了一张 AI 生成的水墨九尾狐封面。

> 📸 截图：首页书架 + 阅读器界面

**功能二：AI 活化——一键生成知识图谱**
这是我最得意的功能。随便选一本书，点「AI 活化」，后台的 DeepSeek 大模型就会把整本书读一遍，自动把里面的人物、地点、概念提取出来，然后画成一张可以拖拽、缩放的网状图。比如《山海经》里的"青丘之山"和"九尾狐"是什么关系，一眼就能看到。

> 📸 截图：知识图谱界面

**功能三：AI 文本分析——粘贴文言文，自动翻译解读**
随便粘贴一段文言文，AI 会帮你做三件事：提取里面的实体（人名、地名等）、翻译成白话文、生成一段深度解读。左边输入，右边出结果。

> 📸 截图：文本分析界面

## 二、创作思路

### 灵感来源

这个想法来自我自己读古籍的痛苦经历。有一次读《山海经》，读到"青丘之山有兽焉，其状如狐而九尾"，我就想：这个青丘之山在哪？九尾狐还出现在哪些故事里？和《聊斋志异》里的狐妖有什么关系？

但传统古籍网站只给你纯文本，这些问题全得自己查。我就在想：**如果 AI 能自动帮我把一本书里所有人物、地点、概念的关系画成一张图，那阅读体验不就完全不一样了吗？**

### 想解决什么问题

- **读不懂**：文言文门槛高，很多人第一眼就放弃了
- **记不住**：一本书几十个人物、几十个地名，读到后面忘了前面
- **看不到关联**：传统阅读是线性的（一页一页翻），但知识是网状的（人物A和人物B有关系，地点C和事件D有关联）

### 为什么选这个方向

1. 大模型特别擅长"实体提取"——从一段文字里识别出人名、地名、概念，古籍里这些关系又很明确，简直是天作之合
2. 市面上的古籍工具要么是纯文本库，要么是扫描件，没有 AI 能力
3. DeepSeek API 便宜、中文理解好，适合做这种垂直场景
4. 选择"社会公益"赛道，因为古籍数字化和文化传承本身就是公益——让更多人能低门槛地接触传统文化

## 四、TRAE 实践过程

### 项目架构

整个项目分前后端两部分：
- **前端**：一个单文件 HTML，没有用 React/Vue 这些框架，纯原生 JS + CSS。包含五个页面：书架、知识图谱、阅读器、文本分析、笔记
- **后端**：Python 写的 Flask 服务，干三件事——提供古籍文本、调 DeepSeek API 做 AI 活化、做文本分析
- **数据**：83 本古籍的纯文本文件，外加 9 本已经活化好的缓存
- **部署**：放在 Railway 上跑，代码推到 GitHub 就自动部署

### 踩坑经历（这才是精华）

**坑一：Vercel 部署直接白屏**

一开始我想用 Vercel 部署（免费嘛），结果死活跑不起来。后来搞明白了原因：我的后端用了 SSE（Server-Sent Events，服务器推送技术，就是服务器可以一点一点地把数据推给浏览器，而不是一次性全给），而 Vercel 是 Serverless 架构（每个请求有时间限制，超时就杀掉），两者天然不兼容。AI 活化一本书要 30-60 秒，Vercel 10 秒就把连接断了。

**解决办法**：换到 Railway（一个云部署平台，支持长时间运行的进程），问题解决。

**坑二：浏览器缓存让我以为代码没生效**

有段时间我改了代码推上去，但打开网页一看——完全没变化。我以为是代码写错了，反复检查，折腾了好久。最后发现是浏览器把旧的网页文件缓存住了，根本没去服务器拿新的。

**解决办法**：在服务器返回网页时加上 `Cache-Control: no-cache` 响应头（就是告诉浏览器"别缓存，每次都来拿最新的"），从此改代码秒生效。

**坑三：封面图被裁剪——"自适应"三个字折腾了我一整天**

这个坑最折磨人。我给《山海经》精选卡片配了张竖版的水墨封面图，但卡片容器是横版的。我用了 `background-size: cover`（CSS 的一个属性，意思是"把图片铺满容器，多出来的部分裁掉"），结果图片上下被裁了，"山海"两个字显示不全。

我试了好几种方案：改容器高度、用 `clamp`（CSS 函数，让尺寸在一个范围内自适应）、改 `aspect-ratio`（宽高比）……都不行。因为只要用 cover，就一定会裁剪。

**最终解决办法**：把 `background-image`（背景图）换成 `<img>` 标签（真正的图片元素），加上 `object-fit: contain`（意思是"完整显示图片，不裁剪，不够的地方留白"）。一行代码的差别，但找原因花了一整天。

**坑四：书架卡片"能点但看不见"**

有用户跟我反馈：书架上的书点得到但看不到。我一开始不信，自己试了一下还真是——鼠标点空白处居然能打开书。

查了半天发现是入场动画的锅。我给卡片设了 `opacity: 0`（透明），然后用一个叫 IntersectionObserver 的浏览器 API 来监听卡片是否进入可视区域，进入 view 后就把透明度变成 1（可见）。问题是这个监听只在页面初始化时绑了一次，如果书架被重新渲染（比如 AI 活化新书后回到首页），新卡片就没有监听，永远停留在透明状态——你看不见但它确实在那。

**解决办法**：不依赖监听，直接在渲染完卡片后用 `setTimeout`（延迟执行）逐个把卡片显示出来。简单粗暴但管用。

**坑五：知识图谱生成到一半切个页面回来就断了**

AI 活化是流式输出的——服务器一点一点把提取的节点推给前端，前端收到一个画一个。问题是我用了 `if(state.page === "graph")` 来判断当前是否在图谱页面，如果用户切到别的页面，这个判断就是 false，新收到的节点就不画了。等用户切回来，进度遮罩也没了，看起来就像"断了"。

**解决办法**：加了一个 `state.isGenerating` 标志位（一个全局变量，记录"是否正在生成"）。切回图谱页面时检查这个标志，如果还在生成就重新显示进度遮罩。而已经收到的节点数据其实都存着，`renderGraph` 函数会重新画出来，不会丢。

**坑六：线上缓存全丢了**

部署到 Railway 后，我挂了一个持久化 Volume（相当于一个不会因为重启而清空的硬盘）到 `/app/cache` 目录，用来保存 AI 活化的结果。但没想到 Volume 是空的，挂载上去后直接把仓库里原有的 9 个缓存文件给覆盖没了。线上显示缓存 0 本。

**解决办法**：把缓存文件复制一份到 `cache_seed` 目录（这个目录不会被 Volume 覆盖），然后在服务器启动时检查：如果 cache 目录文件数比 seed 目录少，就自动从 seed 复制过去。
